## 5. Конфигурация деплоя (amvera.yaml)

Файл `amvera.yaml` размещён в **корне репозитория** (не в `lab3/`) — именно оттуда Amvera читает конфигурацию:

```yaml
meta:
  environment: python
  toolchain:
    name: pip
    version: "3.10"
build:
  requirementsPath: lab3/requirements.txt
run:
  scriptName: lab3/bot.py
  persistenceMount: /data
```

`persistenceMount: /data` — обеспечивает сохранение базы данных `tasks.db` между перезапусками приложения.

## 6. Проблемы и их решения

В процессе деплоя и отладки было выявлено и устранено несколько критических проблем.

---

### 6.1 Проблема: Токен бота не найден при запуске на сервере

**Симптом:** `ValueError: Токен бота не найден` при старте на Amvera, хотя локально бот работал.

**Причина:** На сервере отсутствует файл `.env`. Исходный код использовал только `os.getenv()`, который читает `.env` через `load_dotenv()`, но на Amvera токен передаётся через системные переменные окружения.

**Решение:** Изменён порядок чтения токена — сначала `os.environ.get()` (переменные окружения системы, Amvera), затем fallback на `.env` (локальный запуск):

```python
load_dotenv()  # загружает .env в os.environ, если файл есть
token = os.environ.get("BOT_TOKEN")
BOT_TOKEN: str = token or ""

if not BOT_TOKEN:
    raise ValueError("Токен бота не найден. Укажите BOT_TOKEN в настройках Amvera или в .env")
```

---

### 6.2 Проблема: Бот запускался, но полностью не отвечал (неверный scriptName)

**Симптом:** В логах Amvera появлялась ошибка:
```
python3: can't open file '/app/python3 lab3/bot.py': [Errno 2] No such file or directory
```

**Причина:** В дашборде Amvera поле `scriptName` было заполнено как `python3 lab3/bot.py`. Amvera трактует `scriptName` как путь к файлу и автоматически добавляет `python3` перед ним. В итоге команда запуска выглядела как `python3 "python3 lab3/bot.py"` — файл с таким именем не существует.

**Решение:** В поле `scriptName` указывается только путь к скрипту без интерпретатора: `lab3/bot.py`. Amvera добавляет `python3` сама.

---

### 6.3 Проблема: Неверный конфигурационный файл Amvera

**Симптом:** Amvera использовала старый конфиг `amvera.yml` с параметром `scriptName: app.py`, хотя в репозитории был создан правильный `amvera.yaml`.

**Причина:** В репозитории одновременно существовали два файла:
- `lab3/amvera.yml` (старый, неверный — `scriptName: app.py`, `containerPort: 80`)
- `lab3/amvera.yaml` (новый, верный — `script: python3 lab3/bot.py`)

Amvera читает конфигурацию из **корня репозитория**, а не из подпапок. Старый файл был сгенерирован дашбордом для веб-приложения (с портами), а не для бота.

**Решение:**
1. Удалён `lab3/amvera.yml`
2. Создан `amvera.yaml` в корне репозитория с корректными параметрами

---

### 6.4 Проблема: Утечка SQLite-соединений (бот падал после нескольких запросов)

**Симптом:** Бот отвечал сразу после перезапуска, затем переставал реагировать на команды.

**Причина:** В функции `get_completed_tasks()` в `database.py` использовался вызов без `with`:

```python
# Неправильно — соединение не закрывается!
rows = get_connection().execute("SELECT ...").fetchall()
```

SQLite не допускает несколько одновременных открытых соединений на запись. Накопленные незакрытые соединения приводили к блокировке базы данных.

**Решение:** Все вызовы `get_connection()` приведены к форме с контекстным менеджером:

```python
with get_connection() as conn:
    rows = conn.execute("SELECT ...").fetchall()
```

---

### 6.5 Проблема: Некорректные пути к файлам при запуске из корня репозитория

**Симптом:** `equipment.csv` не импортировался; база данных `tasks.db` создавалась в непредсказуемом месте.

**Причина:** При запуске `python3 lab3/bot.py` из корня репозитория рабочая директория — корень, а не `lab3/`. Пути `"equipment.csv"` и `"tasks.db"` указывали на несуществующие файлы в корне.

**Решение:** Все пути в `database.py` вычисляются относительно расположения самого файла через `__file__`:

```python
_HERE = os.path.dirname(os.path.abspath(__file__))
DB_PATH = "/data/tasks.db" if os.path.exists("/data") else os.path.join(_HERE, "tasks.db")

def _import_csv_if_empty(csv_path: str = os.path.join(_HERE, "equipment.csv")) -> None:
    ...
```

Аналогично в `bot.py` добавлено явное включение папки `lab3/` в `sys.path`:

```python
sys.path.insert(0, os.path.dirname(os.path.abspath(__file__)))
```

### 6.6 Проблема: Дедлок HTTP-соединений — бот получал сообщения, но не отвечал (главная проблема)

**Симптом:** В логах видно `UPDATE RECEIVED` и `cmd_start вызван`, но `sendMessage` никогда не выполнялся — бот зависал на отправке ответа.

**Причина:** Один и тот же экземпляр `HTTPXRequest` использовался для двух разных операций:
- `getUpdates` (long polling) — держит HTTP-соединение открытым ~10 секунд
- `sendMessage` — пытается занять соединение из того же пула

Пул соединений оказывался заблокирован `getUpdates`, и `sendMessage` ждал вечно — **дедлок**.

```python
# Неправильно — один экземпляр для двух разных задач:
request_config = HTTPXRequest(...)
app = ApplicationBuilder()
    .request(request_config)
    .get_updates_request(request_config)  # тот же объект!
    .build()
```

**Решение:** Созданы **два отдельных** экземпляра `HTTPXRequest`:

```python
request_config = HTTPXRequest(
    connect_timeout=30.0, read_timeout=30.0,
    write_timeout=30.0,   pool_timeout=30.0,
)
get_updates_request = HTTPXRequest(
    connect_timeout=30.0, read_timeout=30.0,
    write_timeout=30.0,   pool_timeout=30.0,
)
app = (
    ApplicationBuilder()
    .token(BOT_TOKEN)
    .request(request_config)
    .get_updates_request(get_updates_request)
    .build()
)
```

После этого исправления бот начал стабильно отвечать на все команды.

### 6.7 Проблема: Накопившаяся очередь старых обновлений

**Симптом:** При перезапуске бот сразу получал лавину старых команд, одна из которых могла вызывать ошибку и ронять бот до обработки новых.

**Решение:** Добавлен параметр `drop_pending_updates=True` в `run_polling()`:

```python
app.run_polling(allowed_updates=Update.ALL_TYPES, drop_pending_updates=True)
```

При каждом запуске все необработанные сообщения, накопленные за время простоя, сбрасываются.

## 7. Добавленные улучшения надёжности

| Улучшение | Описание |
|---|---|
| Глобальный error handler | Перехватывает все необработанные исключения, логирует с traceback, уведомляет пользователя |
| `drop_pending_updates=True` | Сброс очереди при старте — бот не обрабатывает устаревшие команды |
| Два экземпляра HTTPXRequest | Раздельные пулы соединений для polling и отправки |
| Пути через `__file__` | Корректная работа при запуске из любой директории |
| `with get_connection()` | Гарантированное закрытие SQLite-соединений |

## 8. Промпт-инжиниринг

В процессе отладки применялись итеративные диагностические промпты:

> «Найди в lab3/bot.py место, где берётся токен, и замени на надёжный вариант с os.environ.get()»

> «Проверь, чтобы каждый вызов get_connection() использовал оператор with»

> «В функции main() поставь /start самым первым в списке add_handler»

Ключевой диагностический шаг — добавление `TypeHandler` с `group=-1` для логирования всех входящих обновлений до их обработки, что позволило локализовать дедлок соединений.
