## 1. Описание интеграции

В рамках работы реализована комбинированная система хранения данных:

* **CSV (equipment.csv):** Используется как первичный источник (справочник) оборудования. При первом запуске бота данные автоматически импортируются в SQLite.
* **SQLite (tasks.db):** Используется для постоянного хранения задач и расширенного списка оборудования, добавленного пользователями. Данные сохраняются между перезапусками.

Архитектура позволяет легко обновлять базовый справочник через CSV, не теряя пользовательские записи в БД.

## 2. Промпт-инжиниринг

**Финальный промпт:**
> "Улучши бота: добавь чтение справочника из equipment.csv, возможность привязки задач к ID оборудования и функцию добавления новых единиц оборудования через интерфейс бота с сохранением в БД."

**Итеративные уточнения:**
- Добавлена возможность добавлять оборудование прямо в диалоге `/add` (кнопка «➕ Новое оборудование»), не прерывая создание задачи.
- Реализован отдельный диалог `/addequipment` для управления справочником независимо от задач.
- Исправлено обращение к полям `sqlite3.Row` через `item["field"]` вместо `item.get("field")`.

### Схема базы данных

**Таблица `tasks`** (расширена по сравнению с Лаб. №1):
| Поле | Тип | Описание |
|---|---|---|
| `id` | INTEGER | Первичный ключ |
| `user_id` | INTEGER | Telegram ID пользователя |
| `title` | TEXT | Текст задачи |
| `priority` | TEXT | Низкий / Средний / Высокий |
| `remind_at` | TEXT | Дата напоминания |
| `completed` | INTEGER | 0 — активна, 1 — выполнена |
| `created_at` | TEXT | Дата создания |
| `equipment_id` | TEXT | **Новое:** FK на таблицу equipment |

**Таблица `equipment`** (новая):
| Поле | Тип | Описание |
|---|---|---|
| `id` | INTEGER | Первичный ключ (автоинкремент) |
| `name` | TEXT | Название агрегата |
| `department` | TEXT | Отдел / цех |
| `status` | TEXT | В работе / Ремонт / Ожидание / Консервация |
| `last_maintenance` | TEXT | Дата последнего ТО |
| `source` | TEXT | `csv` — из файла, `user` — добавлено пользователем |

Связь: `tasks.equipment_id → equipment.id` («Многие к одному»: одно оборудование — много задач).

### Ключевые технические решения

* **Импорт данных:** При первом запуске `init_db()` проверяет количество записей в таблице `equipment`. Если таблица пуста — читает `equipment.csv` через `csv.DictReader` и вставляет строки с `source='csv'`.
* **Связь данных:** Таблица `tasks` расширена полем `equipment_id` (миграция через `ALTER TABLE ... ADD COLUMN` с обработкой `OperationalError` для существующих БД).
* **Два способа добавления оборудования:** через `/addequipment` (3 шага: имя → отдел → статус) и встроенно в `/add` (2 шага: имя → отдел, статус = «В работе»).
* **Обработка ошибок:** Проверка длины ввода, обработка исключений при импорте CSV, корректная работа при отсутствии привязки к оборудованию (`equipment_id = NULL`).

## 4. Новые команды (Лаб. №2)

| Команда | Описание |
|---|---|
| `/equipment` | Полный список оборудования из БД (CSV + пользовательские) |
| `/addequipment` | Добавить новый агрегат в справочник (диалог 3 шага) |
| `/search текст` | Поиск по названию или отделу (регистронезависимо) |
| `/add` | Обновлён: теперь включает шаг выбора оборудования |

## 5. Демонстрация

* **Команда `/equipment`:** Выводит актуальный список из базы данных с эмодзи-статусами.
* **Команда `/addequipment`:** Диалог добавления нового агрегата с выбором статуса через Inline-кнопки.
* **Команда `/add`:** При создании задачи предлагает список оборудования, кнопку «➕ Новое оборудование» для добавления на лету и «⏭ Пропустить».
* **Команда `/search`:** Полнотекстовый поиск по БД.
